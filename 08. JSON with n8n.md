# Module 8: JSON with n8n

> Status: ✅ Complete

---

## Quick Recap

This module goes deep into **JSON** (the format all data travels in inside n8n) and the **Workspace tools** you use to work with it — data types, the Edit Fields node, expression syntax, and pin data.

---

## Part 1: What is JSON, really?

Think of JSON like a **contact card**:
```
Name: John Doe
Age: 30
Has Children: Yes
```

JSON is just the computer-readable version of writing that same info down, in a strict format every app understands the same way:
```json
{
  "name": "John Doe",
  "age": 30,
  "has_children": true
}
```

**The rule that never breaks:** Every JSON is built from **key : value pairs**.
- **Key** = the label, always in double quotes → `"name"`
- **Value** = the actual data → `"John Doe"`

---

## Part 2: The 6 JSON Data Types (memorize these)

| Type | Example | Notes |
|---|---|---|
| **String** | `"Mayank"` | Text — always in double quotes |
| **Number** | `30` | Whole number or decimal — NO quotes |
| **Boolean** | `true` / `false` | Just yes/no |
| **Null** | `null` | Means "nothing/empty" |
| **Array** | `["red", "blue"]` | A list — needs square brackets `[ ]` |
| **Object** | `{"city": "Patna"}` | A JSON *inside* a JSON — needs curly braces `{ }` |

**Common mistake**: forgetting the square brackets `[ ]` for an array, or missing double quotes for a string → n8n immediately throws an error. Syntax has to be exact — same as any programming language.

---

## Part 3: The "Edit Fields" node (also called "Set" node)

This is your **data adder/enricher tool**. Its whole job is: *"take what came in, and add/change some key-value pairs before passing it on."*

Two ways to use it:
1. **Manual mapping** — click "Add Field", type key name, pick type (string/number/etc.), type value
2. **Direct JSON mode** — just write the raw JSON yourself

Both do the exact same thing — just different comfort levels (drag-and-drop vs typing).

---

## Part 4: Expression Syntax (how nodes "reach back" for data)

You've been drag-and-dropping fields — this is what that drag-and-drop actually writes under the hood:

```
{{ $('Node Name').item.json.keyName }}
```

Breaking it down like a sentence:
- `{{ }}` → "this is a dynamic expression, not fixed text"
- `$('Node Name')` → "go fetch data from this specific node"
- `.item.json` → "look inside that node's JSON output"
- `.keyName` → "grab this specific key's value"

**Why this matters**: Once you know the syntax, you don't have to drag-and-drop anymore — you can type expressions directly. This becomes essential once workflows get complex (30-40 nodes) and dragging gets messy.

**Practice example**: To pull a field called `email` from a node named `"Get Lead"`:
```
{{ $('Get Lead').item.json.email }}
```

---

## Part 5: Bonus Concept — "Pin Data"

When testing, you don't want to keep re-triggering real emails/forms every time you tweak a node. **Pinning** freezes a node's output data — so downstream nodes always see that same "frozen" test data, no matter how many times you re-run the workflow.

Super useful for debugging — saves you from spamming real data sources (like sending 20 test emails) while testing.

---

## Part 6: Input/Output Views

Every node has 3 ways to *view* its data — all showing the same thing, just formatted differently:

| View | Best for |
|---|---|
| **Schema** | Quick visual scan of what fields exist |
| **Table** | Spreadsheet-style, good for multiple rows |
| **JSON** | Raw, exact format — use this when writing expressions |

---

## Key Takeaway

- JSON = key-value pairs, has 6 data types: String, Number, Boolean, Null, Array, Object
- The **Edit Fields (Set) node** lets you add/change data before passing it to the next node
- Expression syntax: `{{ $('Node Name').item.json.keyName }}` — this is what drag-and-drop writes automatically
- **Pin Data** freezes test data so you don't keep spamming real APIs/emails while debugging
- Use the **JSON view** when writing expressions, **Schema/Table view** for quick checks
