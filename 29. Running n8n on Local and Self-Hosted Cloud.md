## Overview

This module shows the practical ways to run n8n: (1) n8n Cloud (free trial), (2) Local via Docker, and (3) Self‑hosted on cloud providers (Render, Railway, DigitalOcean, Heroku, AWS). You’ll learn tradeoffs, step‑by‑step commands, and production tips.

---

## 1) n8n Cloud (Quick Start)

- Best for: fast exploration, small proofs of concept, or teams that prefer managed hosting.
- Key benefit: simple signup, 14‑day free trial (no credit card in many cases), built‑in UI and integrations.
- Limitation: tier limits, no STDIO/local MCP, reduced control over underlying host and persistence options.

Practical steps:

1. Sign up for an n8n Cloud trial and open your instance.
2. Create and test workflows in the editor; the cloud gives test and production endpoints for webhooks.
3. When ready to scale or require advanced integrations (STDIO, local MCP), migrate to self‑hosted.

---

## 2) Local via Docker (Recommended for learning)

- Best for: learning, development, or running a personal instance without cloud costs.
- Key benefit: full control, no trial limits, easy to persist data using volumes.
- Requirement: Docker installed on your machine.

Quick `docker` command (development):

```bash
docker run -it --name n8n -p 5678:5678 \
	-v ~/.n8n:/home/node/.n8n \
	n8nio/n8n:latest
```

Environment variables you may want to set (examples):

```bash
docker run -it --name n8n -p 5678:5678 \
	-e N8N_BASIC_AUTH_ACTIVE=true \
	-e N8N_BASIC_AUTH_USER=admin \
	-e N8N_BASIC_AUTH_PASSWORD=strongpassword \
	-v ~/.n8n:/home/node/.n8n \
	n8nio/n8n:latest
```

Docker Compose (simple, persistent example):

```yaml
version: '3.7'
services:
	n8n:
		image: n8nio/n8n:latest
		ports:
			- '5678:5678'
		volumes:
			- ~/.n8n:/home/node/.n8n
		environment:
			- N8N_BASIC_AUTH_ACTIVE=true
			- N8N_BASIC_AUTH_USER=admin
			- N8N_BASIC_AUTH_PASSWORD=strongpassword
```

Notes:

- Use a named volume or host path for persistence (`~/.n8n` or `/var/lib/n8n`).
- Set up basic auth and/or reverse proxy (Nginx/Caddy) with TLS for safe access if exposing to the internet.
- For low‑spec laptops, consider using cloud or lightweight VM if Docker is too heavy.

## 3) Self‑Hosted on Cloud Providers

- Best for: production workloads, team usage, uptime and scaling needs.
- Common hosts: Render, Railway, DigitalOcean, AWS (EC2/ECS), Hostinger, Heroku.

General deployment patterns:

- Deploy using the official Docker image (pull from Docker Hub) or use the provider’s container service.
- Provide environment variables for authentication, external database (Postgres), and file storage.
- Use persistent volumes or external storage for workflows and credentials; consider managed Postgres for reliability.

Render / Railway workflow (high‑level):

1. Create a new Web Service on Render or a Service on Railway.
2. Use the official `n8nio/n8n` image (or your custom image) and set the port mapping.
3. Add required env vars and secrets (basic auth, DB credentials, webhook URL host).
4. Deploy and monitor logs; configure domain and TLS via the host panel.

Important production considerations:

- Use a managed Postgres instead of SQLite for multi‑instance and durability (`DB_TYPE=postgresdb`).
- Run multiple instances behind a load balancer only when using an external database and shared storage for credentials.
- Configure backups for DB and file storage.
- Set resource limits and autoscaling rules as appropriate for your usage.

## Practical Tips & Troubleshooting

- If local Docker fails on low‑spec machines, use a small cloud VM or Render free/web service for lightweight hosting.
- Logs: always check container/host logs for startup errors and permission issues.
- Ports and localhost: when deploying to cloud, remember `localhost` in logs refers to the host machine; use the provider’s public URL.
- Free tiers often spin down after inactivity — use this for testing but not for production.
- Secure credentials: store API keys and tokens in the provider’s secret manager or use `.env` on self‑hosted setups.
