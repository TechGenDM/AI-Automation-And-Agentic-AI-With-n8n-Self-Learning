## Quick Recap

This project builds an **AI code reviewer** — a workflow that automatically reviews every Pull Request (PR) on GitHub and posts helpful comments, just like a senior developer would, but instantly and for free. This is basically a mini version of a real commercial product called **CodeRabbit**.

---

## Part 1: Problem Statement

### What is a Pull Request (PR)?
When working on code in a team, you don't just directly change the main codebase. Instead:
1. You make your changes on a separate "branch" (your own copy)
2. You then create a **Pull Request (PR)** — basically saying "hey, please review and merge my changes into the main code"
3. Someone (a manager/tech lead) reviews it, leaves comments, and either approves or asks for changes

### The real problem being solved
In a company, whenever someone makes a change:
- A manager/tech lead has to manually read every file, check the logic, check naming, check documentation, etc.
- This takes a lot of time — especially for freshers/interns who make more PRs needing review

> **The idea**: let AI do the first ~80% of this review automatically — check logic, naming, structure — so the manager only has to focus on the truly important, complex decisions instead of everything.

### Real-world proof this works
This exact concept is a **paid commercial product** already — **CodeRabbit**. It creates a PR, checks the differences, and posts AI comments — literally the same thing this project builds, just without a subscription fee.

## Part 2: The Workflow — Step-by-Step Breakdown

### The full pipeline
```
PR Trigger (GitHub) → Get File Diffs (HTTP Request) → Create Prompt (Code node) 
→ AI Code Review Agent → Post Comment on GitHub → Add "ReviewedByAI" Label
```

### Node-by-node explanation

| # | Node | What it does |
|---|---|---|
| 1 | **GitHub Trigger (PR Trigger)** | Starts the workflow whenever a pull request event happens on a specified repository (real-time, event-based automation) |
| 2 | **Get File's Diffs from PR** (HTTP Request) | Calls GitHub's own API to fetch the list of changed files and their exact code differences ("diffs") for that PR |
| 3 | **Create Target Prompt from PR Diffs** (Code node) | A JavaScript snippet that formats all the file diffs into one clean, structured prompt for the AI — e.g. *"You are a senior developer. Please review the following code changes..."* |
| 4 | **Code Review Agent** (AI Agent) | The actual AI brain — reads the prompt, reviews the code changes, and generates comments |
| 5 | **GitHub Robot** (GitHub node) | Posts the AI-generated review back as a comment directly on the Pull Request |
| 6 | **Add Label to PR** (GitHub node) | Adds a label like `"ReviewedByAI"` to the PR once the review is posted — optional but useful for tracking |

### The HTTP Request URL (worth understanding)
```
https://api.github.com/repos/{{$json.body.sender.login}}/{{$json.body.repository.name}}/pulls/{{$json.body.number}}/files
```
This is a great example of **dynamic expressions** at work — the username, repository name, and PR number are all pulled automatically from the trigger's data (`$json.body...`), not hardcoded. So this same workflow works for ANY PR on the repo, automatically.

### The Code node's job (in plain terms)
It loops through all the changed files, and for each one:
- Adds the filename
- Adds the actual code diff (the lines that changed)
- Wraps it all into one clear instruction for the AI, ending with something like:
  > *"Review the proposed code changes file by file. Generate inline comments on the relevant lines. Ignore files without patches. Write comments directly, without introducing context."*

This is basically **prompt engineering** — carefully telling the AI exactly what output format you want.

## Part 3: Fixing the GitHub Trigger Node (Real-World Debugging Lesson)

This part of the module is a genuinely valuable lesson: **even official templates can break.**

### The problem
The pre-built "PR Review" template available inside n8n's own template library had a bug — it threw this error:
```
Cannot read properties of undefined (reading 'HTTP code')
```
This happened because GitHub or n8n had recently updated something, and the old template's GitHub Trigger node was no longer compatible.

### How it was diagnosed
The instructor searched the error message online, found the actual GitHub issue/PR on n8n's own GitHub repo where other users had already reported and partially fixed this exact bug.

### The actual fix
Instead of relying on the broken pre-built template:
1. Create a **brand new empty workflow**
2. Manually add a **GitHub Trigger node**
3. Set the event to **"Pull Request"** specifically (not the generic broken trigger setup)
4. Authenticate fresh via **OAuth2** (Connect My Account → authorize on GitHub)
5. Copy-paste the rest of the working nodes (from the provided `.json` file) into this new, correctly-triggered workflow

> **Lesson worth remembering**: No-code tools are still software — they can break with updates. Knowing how to read an error message, search for it, and rebuild the broken piece (instead of panicking) is a real, transferable skill.

### Authentication setup (OAuth2 for GitHub)
- Go to the credential → choose **OAuth2**
- Click **"Connect my account"**
- A GitHub popup appears asking to authorize n8n's access to your account
- Once authorized, you select:
  - **Repository Owner** (your GitHub username — must be typed exactly, it's not a dropdown)
  - **Repository name** (the specific repo to watch)

> **Note**: OAuth2 is preferred here over a raw access token because it's simpler to set up and manage — same idea as the "Predefined Credential" concept from the Auth module.

## Part 4: Running & Testing the Workflow

### Testing each node individually
Before going live, each node was tested one at a time:
1. **Get File Diffs** — confirmed it correctly fetched the changed files (e.g., a `README.md` file with an added line)
2. **Create Target Prompt** — confirmed the JSON output had a clean, well-formatted prompt string ready for the AI
3. **AI Agent** — ran it manually and got back a real code review comment, e.g.:
   > *"This message is a good start, but it could be improved by adding more details... consider mentioning installation instructions or a brief overview."*

### Improving the AI's accuracy with extra context
When the AI's review was slightly generic, more context was added directly into the prompt (e.g., "this repository has to do with FastAPI, not MCP") — and the output immediately became more accurate and specific.

> **This reinforces a very important lesson from earlier modules: the quality of AI output depends heavily on the quality and specificity of the prompt/context you give it.**

### Going live
Once every node worked correctly:
1. The workflow was toggled **Active**
2. A **real PR was created** on GitHub
3. Within seconds, the workflow automatically triggered, generated a review, and posted it as a comment directly on the PR — with zero manual action needed

## Part 5: Live Demonstration (Real Code Example)

A real coding example was used to prove this works on actual, meaningful code changes — not just a README:

- A new API endpoint was added to a FastAPI project
- A `print("Hello World")` debug line was left in the code (a common mistake)
- A new class was added, missing a required base structure (Pydantic model)

**The AI agent automatically caught all of these issues**, for example:
- *"This is a significant addition to the API, but the naming of the endpoint could be improved for better RESTful structure and clarity."*
- *"The addition of 'Hello World' is unnecessary for production code — it can clutter logs. Consider using structured logging."*
- *"The error class is not defined as a Pydantic model, which should be the case."*

> **The real-world value being demonstrated**: even a complete beginner or non-technical manager can now get a first-pass, genuinely useful code review — instantly, automatically, and for free — the moment a PR is created.

## Key Takeaway

- This project = a **free, self-built version of CodeRabbit** — an AI that automatically reviews every GitHub Pull Request
- **Pipeline**: GitHub Trigger (PR event) → Get File Diffs (HTTP Request) → Build Prompt (Code node) → AI Agent reviews → Post comment back to GitHub → Add a label
- **Dynamic expressions** in the HTTP Request URL make the workflow work for any PR automatically, without hardcoding
- **Templates can break** — when official pre-built templates fail, know how to read the error, search for the fix, and rebuild manually
- **Prompt quality matters** — giving the AI more specific context (e.g., "this project uses FastAPI") noticeably improves the quality of its review
- Once **Active**, the workflow runs automatically and instantly, the moment any real PR is created — true "set it and forget it" automation
