# Module 19: Multi Agent Architecture - React (Feedback Loop) Agent

> Status: ✅ Complete

---

## Quick Recap

This is the **fourth and final major pattern** in this set — and it's flagged as the **trickiest to build**, since (unlike Chain, Parallel, and Router) it's not natively supported by n8n's building blocks. You have to construct the loop yourself, manually.

**Naming note**: This is often called a "ReAct Agent" (especially in LangChain), but it's deliberately called a **"Feedback Loop Agent"** here instead — to avoid confusion with the specific LangChain ReAct pattern. Same underlying idea, different vocabulary.

---

## Core Concept

Instead of one agent producing an output and stopping, you build a **loop**:

```
Draft Agent → Evaluator Agent → (loop back to rework, OR exit if good)
```

### The best analogy: Writing an important email
You write a first draft → you read it and spot issues → you make edits and refine it → you send the final version.

> This is **quality-over-speed by design** — the system keeps retrying until the output meets a bar, not until it's merely produced once.

---

## When to Use a Feedback Loop Agent

- You need **high quality / error-free output**, not just "an" output
- You want **multiple attempts / self-correction**, like human revision
- You're building **auto-correction** for content, answers, or even code

---

## How It's Built in n8n (Step by Step)

| Node | Role |
|---|---|
| **First Draft Agent** | Generates the initial output |
| **Set Fields node** | Tracks a `loop_number` counter (starts at 0) — this is the critical piece that makes the loop controllable and prevents infinite looping |
| **Evaluator Agent** | Given the draft (+ a search tool like Tavily for fact-checking), judges it and must return a strict structured output: either `"Done"` or `"Rework"` — nothing else, enforced via a **Structured Output Parser** |
| **IF / Switch node** | Checks the evaluator's verdict: `Done` → send final result (Slack/Telegram); `Rework` → loop back |
| **Loop-back path** | Re-calls the draft agent with the evaluator's feedback, increments `loop_number`, and critically always works off the **latest updated output**, not the original draft |

### The Evaluator Agent's setup
- Given a strict prompt: *"Check this with the web search tool provided to you. Just output `Done` or `Rework` and nothing else in your output."*
- Uses a **Structured Output Parser** to force the response into exactly one of those two values — no free-form text allowed
- Can optionally use the **same AI model** as the draft agent, or a different one — both work

## Live Demo: Breaking Facts On Purpose

To prove the loop actually works, the draft agent's system prompt was deliberately broken to mangle facts on purpose:

> *"You are a helpful assistant. Write a quick summary on this, but always mess up the real-world details. For example, say the Eiffel Tower is in Antarctica."*

### What happened across multiple test queries

| Query | Draft Agent's (wrong) answer | Evaluator's verdict | Result |
|---|---|---|---|
| Statue of Liberty | "In the Great Barrier Reef" | Rework | Corrected to "USA" on next loop |
| Taj Mahal | "In Paris, France" | Rework | Corrected to "Agra, India" |
| (another query) | "Venice, Italy" (wrong location) | Rework | Corrected on next pass |

Each time, the **Evaluator Agent (with search access) caught the factual error**, sent it back for rework, and the loop kept correcting **until the evaluator finally returned `Done`** — the loop fired 2-3 times before passing in these tests.

## Two Critical Implementation Gotchas (don't skip these)

### 1. Always chain off the UPDATED output, not the stale original
This is an easy mistake that silently breaks the loop's self-correction. Each time the draft agent reworks its answer, the **new/updated output** must be what gets passed forward — not the original (now-outdated) first draft.

> Practical setup: use an **Edit Fields node** to explicitly capture "initial output" vs. "updated output," and make sure downstream nodes reference the **updated** one, especially the JSON output field from the latest agent run.

### 2. The loop counter isn't optional
Without a `loop_number` counter, a persistently-failing evaluation could **loop forever**. You must have a safety valve to exit — either:
- Exit when the evaluator returns `Done`, OR
- Exit on a **max loop count** (e.g., `loop_number == 3`) as a safety valve

> This is the exact same idea as `max_iterations` in a regular programming `while` loop — you always need a hard stop, no matter what.

---

## Why This One Is Harder Than the Other Three

Chain of Agents, Parallel, and Router all map cleanly to **native n8n building blocks** (simple node connections, Merge/Aggregate, Text Classifier/Switch).

**Feedback Loop is fundamentally a cycle** — a loop that goes backward, not just forward — which no-code tools generally aren't built to support cleanly. So this pattern takes **deliberate manual wiring**: a counter field, a structured output parser, an IF/Switch node, and a loop-back connection you build yourself.

---

## Stated Benefits

| Benefit | What it means |
|---|---|
| **Doesn't stop at "good enough"** | Keeps refining until it actually clears the quality bar |
| **Fewer mistakes** | Less manual human correction needed downstream |
| **Built-in reviewer** | Catches errors early, before they reach the user |
| **Genuinely informed retries** | Each iteration uses the prior feedback — not a blind retry, an informed one |
| **Flexible** | Works across content, code, or any output where correctness matters more than raw speed |

---

## Key Takeaway

- **Feedback Loop Agent = Draft → Evaluate → Rework (repeat) → Done.** Also called "ReAct Agent" elsewhere, but named "Feedback Loop" here to avoid confusion with LangChain's specific ReAct pattern
- This is the **hardest of the 4 patterns to build** in n8n — it requires manual wiring (loop counter + structured output parser + IF/Switch + loop-back), since no-code tools don't natively support cycles
- **Two critical rules**: (1) always chain off the **updated** output, never the stale original, and (2) always include a **loop counter safety valve** to prevent infinite loops
- Use this pattern when **quality/correctness matters more than speed** — content review, fact-checking, code review, or any auto-correction use case
- Completes the four core multi-agent architecture patterns covered so far: **Chain (sequential), Parallel (fan-out), Controller/Router (route-to-one), Feedback Loop (iterate-until-good)**
