## Overview

RAG (Retrieval Augmented Generation) is a technique that lets LLMs answer questions using fresh, external data sources without retraining.

This module explains the problem RAG solves, the vector embedding concept, the two-step RAG pipeline, and how to visualize the entire flow.

---

## The Problem RAG Solves

LLMs are trained on a fixed dataset up to a **cutoff date**.

Beyond that date, they have no knowledge.

**Example:** If an LLM was trained before a new president took office, asking "Who is the US president?" will return outdated information.

**Two traditional solutions:**

1. Fine-tune the model — retraining weights and parameters (expensive, time-consuming).
2. Provide external context — connect the LLM to a data source so it can look up fresh information (this is RAG).

RAG is the practical modern approach.

---

## The RAG Analogy: Studying for Exams

Think of how you study for a test:

- You have textbooks and course materials (raw data).
- You create notes and summaries (preprocessing and chunking).
- You organize notes by topic so similar chapters are grouped together (embeddings and vector DB).
- When an exam question arrives, you find relevant notes and reference them to write the answer (retrieval and generation).

RAG works the same way:

- Raw documents are broken into chunks.
- Chunks are converted to embeddings (mathematical representations).
- Similar content clusters together in vector space.
- When a query arrives, you find nearby chunks and give them to the LLM as context.

---

## Vector Embeddings: The Core Concept

An **embedding** is a conversion of text into a list of numbers (a vector) in multidimensional space.

**Key property:** Similar concepts are positioned close together in this space.

**Visual example:**

- "Laptop" and "MacBook" are positioned near each other.
- "Ravindra Jadeja" (an all-rounder cricketer) is positioned near other all-rounders.
- "Physics" is far from "Basketball" in embedding space.

**Why this matters:**

- When your query is converted to an embedding using the **same model**, it lands in the same space.
- You can then find the nearest chunks (k-nearest neighbors) and use them as context.

## The Two-Step RAG Pipeline

Every RAG system has **two distinct phases**:

### Step 1: RAG Pipeline (Ingestion)

This is the data preparation phase:

1. **Input:** Raw documents (PDFs, text, websites, etc.)
2. **Chunking:** Break large documents into smaller, manageable pieces (e.g., paragraphs or sections).
3. **Embedding:** Pass each chunk through an embedding model to convert text → numbers.
4. **Vector DB:** Store all embeddings in a vector database (e.g., Pinecone, Weaviate, FAISS, Milvus).

Result: A searchable database of chunks in vector space.

### Step 2: RAG Agent (Retrieval & Generation)

This is the query-answer phase:

1. **Query input:** User asks a question.
2. **Embed query:** Convert the query to an embedding using the **same embedding model**.
3. **Search:** Find the k nearest chunks in the vector DB (k-nearest neighbor search).
4. **Context assembly:** Gather the top chunks as context.
5. **LLM call:** Pass context + query to the LLM.
6. **Response:** The LLM generates an answer informed by the retrieved context.

---

## Critical: Same Embedding Model

**This is the most important rule:** You must use the **same embedding model** for both:

- Creating embeddings during ingestion (Step 1)
- Embedding the query during retrieval (Step 2)

**Why?**

- If you use different models, embeddings won't align in the same space.
- Your query will land in the wrong region of vector space.
- You'll retrieve irrelevant chunks.

**Analogy:** If you write exam notes in English but the exam is in Spanish, you won't find the right answers.

## Practical RAG Workflow

### 1. Data Preparation

- Collect documents (text files, PDFs, web content, databases).
- Extract and clean the text.
- Divide into chunks (typical: 256–512 tokens per chunk, with overlap to preserve context).

### 2. Embedding & Storage

- Use an embedding model (OpenAI `text-embedding-3-small`, `sentence-transformers`, etc.).
- Convert each chunk to a vector.
- Store in a vector DB with metadata (source, timestamp, etc.).

### 3. Query Time

- Accept user query.
- Embed the query using the **same model**.
- Search the vector DB for the top-k similar chunks (default k=5–10).
- Assemble context from retrieved chunks.
- Pass context + query to LLM: `"Context: [chunks]. Question: [query]. Answer:"`
- Return LLM response.

---

## Why RAG Works

1. **Up-to-date information:** External sources can be updated constantly without retraining.
2. **Domain-specific knowledge:** Embed internal documents, product manuals, policies, etc.
3. **Reduced hallucination:** LLM answers are grounded in retrieved facts.
4. **Efficient:** No retraining needed; just update the vector DB.

---

## Key Takeaways

- RAG = Retrieval (find relevant data) + Augmented (give context to LLM) + Generation (LLM produces answer).
- Two phases: **(1) Ingestion** (documents → embeddings → vector DB) and **(2) Retrieval** (query → search → LLM).
- Vector embeddings map similar concepts close together in multidimensional space.
- **Always use the same embedding model** for ingestion and retrieval.
- RAG lets LLMs answer questions about data they were never trained on.

## Real-World Use Cases

- **Customer support chatbots:** Retrieve from product docs, FAQs, tickets.
- **Internal Q&A:** Query company policies, procedures, onboarding docs.
- **Medical/legal:** Augment LLM with case law, medical literature, regulations.
- **News research:** Ground answers in recent articles and reports.
- **Perplexity-style search:** Retrieve web results and summarize for users.
